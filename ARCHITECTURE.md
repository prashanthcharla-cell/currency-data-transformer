## Design Patterns Used

### 1. **Builder Pattern**
- `Transaction.builder()` - Immutable object creation
- `ValidationResult.builder()` - Complex result object
- `ErrorResponse.builder()` - Error response construction

### 2. **Dependency Injection**
- Constructor-based injection with `@RequiredArgsConstructor`
- Spring manages component lifecycle
- Testable with mock dependencies

### 3. **Strategy Pattern**
- Validation logic encapsulated in `CsvTransactionValidator`
- Easily extensible for different validation strategies

### 4. **Exception Handling Pattern**
- Custom exceptions (`InvalidFileException`)
- Centralized handling (`GlobalExceptionHandler`)
- Consistent error responses

### 5. **Result Object Pattern**
- `ValidationResult` encapsulates multiple return values
- Avoids throwing exceptions for business validation
- Provides detailed feedback

### 6. **Single Responsibility Principle**
- Controller: HTTP handling
- Service: Business orchestration
- Validator: Validation logic
- Models: Data representation

## Key Design Decisions

### 1. **Streaming vs Loading**
- âœ… **Chosen**: Streaming with `BufferedReader`
- **Reason**: Memory efficient for large files
- **Trade-off**: Slightly more complex parsing

### 2. **Custom CSV Parser vs Library**
- âœ… **Chosen**: Custom parser
- **Reason**: No external dependencies, simple requirements
- **Trade-off**: More code to maintain

### 3. **Exception vs Result Object**
- âœ… **Chosen**: Result object for row validation
- **Reason**: Collect all errors, not just first one
- **Trade-off**: More complex return type

### 4. **Fail Fast vs Collect All Errors**
- âœ… **Chosen**: Collect all errors
- **Reason**: Better user experience
- **Trade-off**: More processing time

### 5. **Duplicate Headers Handling**
- âœ… **Chosen**: Use first occurrence
- **Reason**: Predictable behavior
- **Trade-off**: Could be confusing if not documented
